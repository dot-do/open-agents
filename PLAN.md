Summary: Evaluate whether a browser-compatible, mosh-like terminal architecture would materially improve Open Harness terminal latency, and recommend the most practical next step. The recommended approach is to keep the PTY in the sandbox but replace the current raw remote-echo bridge with a mosh-style predictive/state-synced transport only if we are willing to build substantial custom infrastructure; otherwise, the practical near-term path is to improve the current browser client and transport while accepting remote RTT.

Context: The current terminal path is a simple browser terminal client embedded in an iframe that sends raw keystrokes over WebSocket to a sandbox PTY and renders returned output. In [apps/web/lib/sandbox/terminal/server-script.ts](#workspace-file=apps/web/lib/sandbox/terminal/server-script.ts), the client does `term.onData((data) => socket.send(data))` and renders server output with `term.write(event.data)`. The server receives each message and calls `ptyProcess.write(message)`, then forwards PTY output back with `ws.send(data)`. The terminal is mounted in an iframe from [apps/web/app/sessions/[sessionId]/chats/[chatId]/terminal-panel.tsx](#workspace-file=apps/web/app/sessions/[sessionId]/chats/[chatId]/terminal-panel.tsx). Ghostty Web’s README describes it as a browser VT100 implementation with xterm.js API compatibility, not a latency-hiding transport. Mosh’s README explicitly says its advantage comes from speculative local echo and predictive terminal-state modeling, including support for full-screen apps like vi/emacs, but Mosh depends on UDP and a native client. A browser-only SSH client would still be remote echo and would not solve the core RTT problem.

Approach: Do not pursue “browser SSH” as the fix. If the goal is dramatically better feel for typing and TUIs like vim/nvim, the only architectural class that really addresses the problem is a browser-compatible mosh-like transport with client-side prediction and server reconciliation. That is a major project, and there does not appear to be an obvious drop-in browser mosh solution to adopt here. Therefore the recommendation is a two-track decision: (1) near term, improve the existing architecture by removing the iframe, embedding Ghostty Web directly in the app, and checking whether the websocket path can be made more direct; (2) if terminal feel is a strategic product priority, scope a custom predictive transport prototype rather than swapping transports to SSH.
